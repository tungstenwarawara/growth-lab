## 3. 大賢者の肥大化問題と解決策

### 問題の本質

大賢者は**5ファイル・1,095行の仕様書**だが、実際に動作したことがない。
これは「作りたいもの」と「今必要なもの」が混同されている。

### 公式ベストプラクティスから見た大賢者の課題

| 課題 | 公式の指針 | 現状 |
|------|----------|------|
| Skill が大きすぎる | 200行以下が目安 | 354行（content-manager） |
| 全部自動で動かそうとしている | `disable-model-invocation: true` で手動制御 | 全部自動前提 |
| 車輪の再発明 | 公式 Memory, Hooks を使う | 独自 YAML で全部やろうとしている |
| コンテキストコスト未考慮 | 機能追加 = 精度低下の可能性 | 「足し算」思考で拡張 |

### 解決策: 大賢者を「3層構造」に再設計

```
Layer 1: コア（常に動く、軽量）
  - SKILL.md: 50行以下。基本的な振る舞いのみ
  - description を明確に書き、自動読み込みの精度を上げる

Layer 2: ワークフロー（/コマンドで呼出、disable-model-invocation: true）
  - /orchestrate: タスク生成・分解
  - /content: コンテンツ管理
  - /mcp-scan: MCP探索

Layer 3: リファレンス（必要時のみ読み込み）
  - プロトコル仕様
  - YAML通信フォーマット
  - 評価基準
```

### 精度を落とさない拡張ルール

1. **1機能追加するなら1つ削る** — 純増させない
2. **200行を超えたら分割する** — 大きいSkillは精度が下がる
3. **まず手動で動かす** → 安定したら自動化
4. **公式機能を最大活用** → 独自実装は差別化部分のみ
5. **コンテキストコストを意識する** → 追加のたびにテスト
